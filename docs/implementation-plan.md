## Цель MVP

Создать упрощенный интернет-магазин с каталогом, корзиной, оформлением заказа и
асинхронным уведомлением о созданном заказе.

Проект должен на практике показать:

- независимые микросервисы и владение данными;
- синхронное HTTP-взаимодействие;
- асинхронное взаимодействие через брокер сообщений;
- кеширование и хранение временных данных;
- наблюдаемость распределенной системы;
- локальную оркестрацию через .NET Aspire;
- интеграционное тестирование инфраструктуры.

## Архитектура MVP

```text
Client / Blazor (позже)
          |
          v
      API Gateway
       /   |    \
      v    v     v
 Catalog  Cart  Orders
    |      |      |
 PostgreSQL Redis PostgreSQL
                  |
                  v
               RabbitMQ
                  |
                  v
         Notifications Worker

 OpenTelemetry -> Prometheus -> Grafana
```

### Границы сервисов

| Компонент | Ответственность | Хранилище |
| --- | --- | --- |
| Gateway | Единая внешняя точка входа и маршрутизация | Нет |
| Catalog | Товары, цены и доступное количество | PostgreSQL |
| Cart | Временные корзины пользователей | Redis |
| Orders | Заказы и снимки позиций на момент покупки | PostgreSQL |
| Notifications | Обработка события создания заказа | Нет на старте |

Каждый сервис владеет своими данными. Общая база данных и прямой доступ одного
сервиса к таблицам другого сервиса не допускаются.

## Ограничения MVP

На первом релизе не реализуются:

- полноценная аутентификация и авторизация;
- настоящие платежи;
- резервирование остатков;
- доставка;
- распределенная saga;
- реальная отправка email;
- Kubernetes и production-развертывание.

До появления Identity-сервиса идентификатор пользователя передается явно через
API. Конкретный временный контракт будет выбран при реализации Gateway.

| Дата | Решение | Причина |
| --- | --- | --- |
| 2026-06-14 | Начать с четырех бизнес-компонентов и Gateway | Сохранить небольшой scope, но показать синхронное и асинхронное взаимодействие |
| 2026-06-14 | База данных принадлежит одному сервису | Не связывать сервисы общей моделью данных |
| 2026-06-14 | Использовать Aspire для локальной оркестрации | Единый запуск приложений, инфраструктуры и телеметрии |
| 2026-06-14 | Отложить Identity, платежи и saga | Сначала завершить основной сценарий оформления заказа |
| 2026-06-14 | Не создавать общий проект со всеми DTO | Избежать сильной связанности сервисов |
| 2026-06-14 | Использовать .NET 10 и Aspire 13.4.3 | Эти версии установлены и поддерживаются актуальными шаблонами Aspire |
| 2026-06-14 | Централизовать версии NuGet-пакетов | Обновлять зависимости согласованно для всего solution |
| 2026-06-14 | Закрепить MessagePack 2.5.301 | Устранить известную уязвимость транзитивной версии 2.5.192 |
| 2026-06-14 | Использовать HTTP для локального AppHost | Исключить HTTPS из локального dashboard-профиля; production transport будет настраиваться отдельно |
| 2026-06-14 | Использовать persistent volumes для PostgreSQL, Redis и RabbitMQ | Не терять локальные данные при пересоздании контейнеров |
| 2026-06-14 | Использовать официальные Aspire client integrations | Получить единые DI-регистрации, telemetry и health checks зависимостей |
| 2026-06-14 | Временно запускать AppHost на `net8.0`, оставив сервисы на `net10.0` | Обойти ошибку TLS `unexpected EOF` между AppHost на .NET 10.0.3 и локальным DCP Aspire 13.4.3 |
| 2026-06-14 | Использовать Docker 29 для локальной разработки | Версия совместима с минимальным требованием Aspire 13.4.3 и успешно запускает весь стек |
