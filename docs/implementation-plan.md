## Цель MVP

Создать упрощенный интернет-магазин с каталогом, корзиной, оформлением заказа и
асинхронным уведомлением о созданном заказе.

Проект должен на практике показать:

- независимые микросервисы и владение данными;
- синхронное HTTP-взаимодействие;
- асинхронное взаимодействие через брокер сообщений;
- кеширование и хранение временных данных;
- наблюдаемость распределенной системы;
- локальную оркестрацию через .NET Aspire;
- интеграционное тестирование инфраструктуры.

## Технологический стек

- .NET 10 и ASP.NET Core
- .NET Aspire
- PostgreSQL
- Redis
- RabbitMQ
- Entity Framework Core и Npgsql
- MassTransit
- YARP
- OpenTelemetry
- Prometheus
- Grafana
- xUnit
- Testcontainers или тестовые ресурсы Aspire
- Blazor на более позднем этапе

Версии пакетов фиксируются при реализации первого этапа и далее обновляются
осознанно, а не автоматически.

## Архитектура MVP

```text
Client / Blazor (позже)
          |
          v
      API Gateway
       /   |    \
      v    v     v
 Catalog  Cart  Orders
    |      |      |
 PostgreSQL Redis PostgreSQL
                  |
                  v
               RabbitMQ
                  |
                  v
         Notifications Worker

 OpenTelemetry -> Prometheus -> Grafana
```

### Границы сервисов

| Компонент | Ответственность | Хранилище |
| --- | --- | --- |
| Gateway | Единая внешняя точка входа и маршрутизация | Нет |
| Catalog | Товары, цены и доступное количество | PostgreSQL |
| Cart | Временные корзины пользователей | Redis |
| Orders | Заказы и снимки позиций на момент покупки | PostgreSQL |
| Notifications | Обработка события создания заказа | Нет на старте |

Каждый сервис владеет своими данными. Общая база данных и прямой доступ одного
сервиса к таблицам другого сервиса не допускаются.

## Ограничения MVP

На первом релизе не реализуются:

- полноценная аутентификация и авторизация;
- настоящие платежи;
- резервирование остатков;
- доставка;
- распределенная saga;
- реальная отправка email;
- Kubernetes и production-развертывание.

До появления Identity-сервиса идентификатор пользователя передается явно через
API. Конкретный временный контракт будет выбран при реализации Gateway.
